## Required fields

`generated_at`, `languages`, `primary_language`, `frameworks`, `entry_points`, `modules`, `organization_suggestion`

## Optional fields

All others; include only what is found.

### Field `signals`

Optional top-level detection signals. Scout uses this for cheap pre-walk facts that other agents or the Reversa orchestrator can react to. Content Server signal types are:

- `cs_agent_probe_failed`
- `cs_agent_no_profile`
- `cs_agent_profile_detected`
- `cs_agent_profile`
- `cs_agent_snapshot_failed`

### Field `content_server`

Optional. Present only when `[integrations.cs_agent].enabled = true` and `.reversa/context/cs-agent/_meta.json` exists. Values are copied from the read-only cs-agent snapshot. The Scout must not populate this field by recursively walking `srcmodules`.

## Field `organization_suggestion`

Suggestion for how to organize the specs for this project. Read by the Reversa orchestrator to pre-select the default option in the spec organization menu.

### Sub-fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `granularity` | string | yes | One of: `module`, `use-case`, `endpoint`, `hybrid`, `feature`, `custom`. The Scout never suggests `custom` — that value comes from the user's choice only. |
| `rationale` | string | yes | Short sentence explaining why this granularity was chosen. Appears in the menu as the Scout's "Reason:". |
| `signals` | array | yes | List of detected signals that led to the suggestion. Each item has `type` and `evidence` (list of paths). May be empty when the `feature` fallback is used. |
| `features` | array | required when `granularity = "feature"` | List of feature names discovered by the Scout. Each name becomes a top-level folder. |

### Heuristics for defining `granularity`

| Detected signal | Suggested `granularity` |
|-----------------|-------------------------|
| Centralized routing (`routes.*`, `urls.py`, `*Controller.cs`, `@RestController`) | `endpoint` |
| Top-level folders with domain names (`auth/`, `orders/`, `payments/`) | `module` |
| Gherkin / behavior-driven E2E specs (`features/*.feature`, `*.spec.*` BDD) | `use-case` |
| Multiple signals coexisting with similar weight | `hybrid` |
| No clear organizational signal | `feature` (fallback; populate `features` with what could be extracted) |

### Immutability

After the first run, the orchestrator persists the suggested `granularity` in `.reversa/config.toml` under the `scout_suggestion` field. On re-runs, the Scout may regenerate `surface.json` (the legacy project may have changed), but the orchestrator does NOT update `scout_suggestion` in `config.toml` (RF-14 of the spec organization specification).

## Note

Use this schema as a guide. If a field does not apply to the project, omit it, except for the required fields listed above.
